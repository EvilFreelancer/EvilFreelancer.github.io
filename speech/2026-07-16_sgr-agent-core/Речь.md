## Слайд 4. Что такое SGR

Начнём с главного. SGR это Schema-Guided Reasoning, рассуждение по схеме.

Слева на слайде обычный агент - чёрный ящик. Запрос ушёл, внутри произошла какая-то магия, ответ пришёл. Что там происходило и почему - нам не видно.

Справа SGR-агент - прозрачный. Мы явно описываем фазы - рассуждение, планирование, действие, ответ. И каждая фаза это не пожелание в промпте, а строгая схема, которую модель обязана заполнить. За счёт этого рассуждение видно, его можно проверить и повторить.

То есть мы не просим модель быть умной, мы даём ей рельсы, показываем как думать, а дальше она уже сама.

## Слайд 5. Схема в коде

Чтобы не звучало абстрактно, вот та самая структура в коде.

Это ReasoningTool, обычная pydantic-модель. После конвертации в JSON-схему она уходит в модель как формат ответа.

Поля - шаги рассуждения, текущая ситуация, статус плана, флаг хватает ли данных, что осталось и завершена ли задача.

Каждое поле это обязательный шаг, а не пожелание. Модель заполняет всю схему целиком, и мы получаем видимое, повторяемое рассуждение прямо в структуре ответа.

## Слайд 6. Архитектура

Теперь про то, как это крутится. Один и тот же агент запускается тремя способами.

Первый - API, это HTTP-сервер с OpenAI-совместимым эндпоинтом. Стриминг по SSE, режимы stateless и stateful, работает с Open WebUI, LibreChat и любым OpenAI-клиентом.

Второй - ACP, Agent Client Protocol поверх stdio, для локальных агентов прямо в редакторах - Obsidian, Zed, VS Code.

Третий - CLI, интерактивный шелл в терминале для одиночных запросов.

Логика агента одна, меняется только способ запуска, а конфиг один и тот же.

## Слайд 7. ACP подробнее

Пара слов про ACP, потому что это относительно новая для многих штука.

Это протокол для локальных агентов поверх stdio. Агент запускается как подпроцесс и общается по JSON-RPC. Поддерживает tools, resources и prompts.

Хосты, которые его понимают - Obsidian с плагином Agent Client, Zed с нативной панелью агента, VS Code и JetBrains через расширения, Claude Desktop.

Запускается командой sgracp с тем же конфигом, что и HTTP-сервер.

## Слайд 8. API Mode

Теперь про серверную часть подробнее. Это FastAPI-сервер, который хранит агентов в памяти и управляет их жизненным циклом.

Цикл простой - создание, то есть загрузка конфига и инициализация тулов, выполнение, то есть обработка запроса через SGR-пайплайн, и очистка.

Два режима. Stateless - клиент шлёт полный контекст каждый раз, как обычный REST. Stateful - диалог по agent_id, история хранится на сервере. Плюс стриминг по SSE и возможность отменить выполнение.

Важно помнить, что stateful-агент живёт в памяти сервера и имеет таймаут.

## Слайд 9. Системные тулы

Из чего собирается агент. Начнём с тулов.

Из коробки идёт готовый набор. Системные - reasoning для анализа и выбора следующего шага, generate_plan и adapt_plan для плана, clarification для уточнений, final_answer и answer для ответов.

Прикладные - web_search с провайдерами Tavily, Brave и Perplexity, extract_page_content для контента страниц, create_report для отчётов с цитатами, и run_command для команд и выполнения кода в песочнице.

Это те кирпичики, из которых собираются агенты.

## Слайд 10. Встроенные агенты

Агенты, которые есть сразу.

sgr_agent - максимально строгий и управляемый SGR. sgr_tool_calling_agent - SGR-логика плюс нативный tool calling, больше совместимости и отказоустойчивости. tool_calling_agent - просто тулколлинг без SGR. dialog_agent - диалоговый, с промежуточными результатами. И iron_agent - упрощённый, без tool calling, для совсем простых сценариев.

Любой можно расширить своим наследником, механизм ядра при этом общий.

## Слайд 11. Конфигурация

Теперь то, ради чего многие и приходят - конфигурация. Ключевая философия SGR это конфигурация вместо кода.

Есть иерархия настроек. Сначала дефолты фреймворка, потом переменные окружения, потом config.yaml, потом параметры командной строки. Каждый следующий слой переопределяет предыдущий.

Сам конфиг разбит на секции - llm, execution, tools, mcp, agents и acp. Дальше пройдёмся по главным реестрам.

## Слайд 12. Реестр тулов

Реестр тулов. Здесь мы регистрируем инструменты и настраиваем их.

Можно просто объявить тул пустым словарём, если он без параметров. Можно задать конфигурацию - например у веб-поиска указать провайдера, ключ и лимиты. А можно подключить свой кастомный тул из модуля через base_class.

Тул, объявленный тут, дальше подключается к агентам по имени.

## Слайд 13. Реестр MCP

Реестр MCP-серверов. MCP это Model Context Protocol, способ подключить внешние источники с их тулами.

Сервер можно подключить по URL, как удалённый deepwiki, или запустить локально как stdio-процесс, например filesystem через npx.

Агент автоматически получает тулы от подключённых MCP-серверов, ядро при этом трогать не нужно.

## Слайд 14. Реестр агентов

И реестр агентов. Тут определяем конкретного агента.

Выбираем базовый класс, при необходимости переопределяем настройки LLM - модель, температуру, и перечисляем список тулов.

Важный момент - для SGR-агента в списке обязательно должен быть reasoning_tool. Именно он запускает фазу рассуждения по схеме, без него агент не будет думать структурно.

Ключевая идея всего блока - минимальные переопределения. Базовый конфиг задаёт LLM и лимиты, а конкретный агент только выбирает класс и тулы.

## Слайд 15. Связь компонентов

Соберём картину целиком.

Наверху агент - это SGR Pipeline с фазами Reasoning, Planning, Acting. Он опирается на слой тулов - зарегистрированные в конфиге инструменты вроде web_search, extract_page_content, create_report, run_command.

А снизу подключаются внешние MCP-серверы - deepwiki, filesystem, git и другие, которые добавляют агенту ещё тулов.

То есть агент, тулы и MCP это три слоя, которые собираются конфигом.

## Слайд 16. Свой инструмент

Теперь как расширять. Начнём с самого нижнего уровня - свой тул.

Тул это схема плюс один метод. Импортируем BaseTool из ядра и Field из pydantic, наследуемся от BaseTool, задаём имя, описываем поля через Field.

Тонкость в том, что поля Field это одновременно и аргументы тула, и подсказка модели, как их заполнить - описание идёт прямо в схему.

А в методе call пишем свою логику и возвращаем текст, который увидит модель. Всё, тул готов, можно подключать в конфиге.

## Слайд 17. Свой агент

Уровень выше - свой агент, когда нужна своя логика.

Наследуемся от готового типа, например SGRToolCallingAgent, задаём имя и список тулов. Для файлового агента добавляем working_directory, который ограничивает песочницу, за неё агент не выйдет. При желании переопределяем шаги цикла.

Наследник регистрируется в ядре автоматически и становится доступен по имени в конфиге, как и встроенные.

## Слайд 18. Как начать

Как это запустить. Ставим пакет через pip install sgr-agent-core.

Копируем пример конфига, прописываем ключи, и дальше один и тот же агент можно поднять в любом режиме.

Команда sgr - HTTP-сервер с OpenAI-совместимым API. Команда sgrsh - интерактивный CLI. Команда sgracp - агент по протоколу ACP через stdio.

После этого обращаемся к нему как к обычному OpenAI API, в поле model указываем имя нашего агента.

## Слайд 19. Пример - Deep Research

Первый готовый пример - Deep Research, агент глубокого исследования темы.

Пайплайн такой. Сначала generate_plan строит план. Потом web_search ищет в интернете. extract_page_content вытаскивает контент страниц. reasoning анализирует и при необходимости адаптирует план. create_report собирает отчёт в Markdown с цитатами. И final_answer выдаёт результат.

Всё это не чёрный ящик - в логах виден каждый шаг, а итоговый отчёт сохраняется в reports.

## Слайд 20. Пример - файловый агент

Второй пример, ради которого многие и приходят - файловый агент, работа с локальными файлами.

Пайплайн проще. reasoning анализирует запрос и планирует. find ищет файлы по имени. read_file и grep читают и анализируют содержимое. final_answer формирует ответ.

По умолчанию всё read-only, а working_directory ограничивает агента одной папкой. Через ACP такого агента удобно подключить прямо к Obsidian и спрашивать по своим заметкам.

## Слайд 21. Куда подключается

Отдельно про то, куда это подключается, потому что свой фронт писать не нужно.

Через OpenAI API работают Open WebUI, LibreChat, Continue и любой клиент, который умеет OPENAI_BASE_URL.

Через ACP - Obsidian с плагином Agent Client, Zed с нативной панелью агента, VS Code и JetBrains через расширения.

Берём готовый клиент и просто указываем адрес нашего агента.

## Слайд 22. Метрики и Langfuse

И про наблюдаемость, потому что агент это не только запустить, но и понять, что он делает.

Каждый шаг виден в логах, это прозрачный пайплайн, а не чёрный ящик. Тулы тестируются как обычные классы - unit-тест на вход и выход.

Для трейсинга есть Langfuse, подключается блоком в конфиге, показывает вызовы, задержки и стоимость.

А метрики простые - полнота ответа, число лишних вызовов тулов, регрессия при смене модели. Совет из практики - проверяйте не только правильный путь, но и как агент ведёт себя на пустых папках и странных запросах.

## Слайд 23. Итог

Давайте резюмируем, зачем всё это.

Предсказуемость - рассуждение идёт по схеме, без пропущенных шагов. Переносимость - любой OpenAI-совместимый провайдер и локальные модели. Расширяемость - свои тулы, схемы и агенты через конфиг и наследование. И экосистема - готовые клиенты через OpenAI API и ACP, MCP и Langfuse.

Проект делает команда neuraldeep и vamplabAI при поддержке red_mad_robot, концепция SGR выросла из сообщества LLM Under the Hood.
